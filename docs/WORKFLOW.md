# 開発ワークフロー

## Claude Code が各フェーズで守る手順

```
1. ブランチを切る
   git checkout -b phase/N-description

2. 設計ドキュメントをファイルに書く（docs/design/phase{N}-*.md）
   ※「設計を提示する」= チャットに書くだけでは不十分。必ずファイル化する。
   内容: 変更ファイル一覧・クラス設計・メソッドシグネチャ・スレッド境界・判断理由

3. 設計を Agent でレビューする（ジュニア視点 + シニア視点）
   → 重大指摘があれば設計を修正して再レビュー

4. 実装する（ブランチ上でコミット）

5. 実装を Agent でレビューする（ジュニア視点 + シニア視点）
   → 重大指摘があれば修正して再レビュー

6. ビルド確認・動作テスト

7. ユーザーに確認を求める

8. main へマージ（--no-ff）
```

**絶対に守ること:**
- 手順 2〜3 をスキップして実装に入らない
- 手順 4〜5 をスキップして「問題ない」と自己判断しない
- レビューは必ず `Agent` ツールでサブエージェントを起動して行う（自分でコードを眺めるだけは不可）

---

各フェーズで以下の 2 サイクルを回す。**指摘が重大でない限り、各サイクルは最大 2 ラウンドで完結させる。**

## Git ブランチ戦略と Worktree 運用

### ブランチ命名規則

```
main                        # フェーズ単位でマージされる安定ライン
phase/1-skeleton            # Phase 1 作業ブランチ
phase/2-vocoder-core        # Track A
phase/3-drawable-carrier    # Track B
phase/4-band-gain           # Track A(Phase 2 完了後)
phase/5-lfo-dsp             # Track C(DrawableCurveComponent 完了後)
phase/5-integration         # Phase 5 統合(Phase 2+3 完了後)
phase/6-polish              # Phase 6
```

### Worktree による並列作業

並列トラックは `git worktree` で独立した作業ディレクトリを持ち、互いに干渉しない。

```sh
# Phase 1 完了・main にマージ後、Track A と B を同時に開始する例
git worktree add ../vocoder-phase2 -b phase/2-vocoder-core
git worktree add ../vocoder-phase3 -b phase/3-drawable-carrier

# Track C の追加(DrawableCurveComponent が main にマージされてから)
git worktree add ../vocoder-phase5-dsp -b phase/5-lfo-dsp

# 作業後の掃除(ブランチを main にマージしてから)
git worktree remove ../vocoder-phase2
```

各 worktree は `vocoder/` の兄弟ディレクトリに配置する(`../vocoder-phaseN/`)。

### Claude Code での worktree 活用

並列 Agent を起動するとき `isolation: "worktree"` を指定すると、Claude Code が worktree の作成・後片付けを自動管理する。

```
Agent(
  isolation: "worktree",    ← 自動で worktree を切って作業
  subagent_type: "claude",
  prompt: "phase/2-vocoder-core ブランチで FilterBank を実装..."
)
```

- 変更がなければ worktree は自動削除される
- 変更があれば完了時にブランチ名とパスが返ってくる → レビュー後に main へマージ

### マージポリシー

```
phase/N  ──[レビュー完了]──► git merge --no-ff phase/N -m "feat: Phase N complete" ──► main
```

- `--no-ff` でフェーズ単位のマージコミットを残す
- 依存フェーズは **main へのマージ完了を確認してから** 後続ブランチを `main` ベースで切る
- `PluginProcessor.{h,cpp}` は複数トラックが変更したい場合、**先に完了したトラックを main にマージしてから** 後続トラックが `git rebase main` で取り込む

### ファイル競合ホットスポット

| ファイル | 複数トラックが触る可能性 | 対策 |
|---|---|---|
| `Source/PluginProcessor.{h,cpp}` | 高 | トラックごとに完了→マージ→次トラックがリベース |
| `Source/state/Parameters.{h,cpp}` | 高(各フェーズがパラメータ追加) | 同上 |
| `CMakeLists.txt` | 中(sources 追加) | sources は `Source/CMakeLists.txt` に分離して各トラックが独立編集 |
| `Source/dsp/*.{h,cpp}` | 低(トラックごとに別ファイル) | 並列編集してよい |
| `Source/gui/*.{h,cpp}` | 低(トラックごとに別ファイル) | 並列編集してよい |

## レビューの実施方法

**すべてのレビューは `Agent` ツールでサブエージェントを起動して行う。**  
自分自身でコードを見て「問題ない」と判断するだけでは不十分。独立した視点を得るために必ずエージェントを立てる。

```
# 設計レビューの例
Agent(
  subagent_type: "claude",
  prompt: """
    以下の設計を2つの視点でレビューしてください。
    [設計内容をここに貼る]

    ## ジュニア視点
    - クラス名・メソッド名が意図を正確に表しているか
    - ...

    ## シニア視点
    - スレッド境界が明確か
    - ...

    各視点で「重大/中程度/軽微」に分類した指摘リストを返してください。
  """
)

# コードレビューの例
Agent(
  subagent_type: "claude",
  prompt: """
    以下のファイルをレビューしてください。
    対象ファイル: Source/dsp/MipmapTable.cpp, Source/dsp/WavetableOscillator.cpp
    [ファイルパスを渡して読ませる]

    ## ジュニア視点 / シニア視点
    ...指摘リストを返してください。
  """
)
```

サブエージェントから返ってきた指摘を確認し、重大なものは修正してから次のステップへ進む。

## サイクル 1: 設計レビュー

```
設計ドラフト作成
    ↓
[ラウンド 1]
  Agent起動(ジュニア視点) → 指摘リスト → 修正
  Agent起動(シニア視点)  → 指摘リスト → 修正
    ↓
[ラウンド 2 ※必要なら]
  同上(未解決 or 新たな重大指摘のみ対象)
    ↓
設計確定 → 実装へ
```

### ジュニア視点でチェックすること
- クラス名・メソッド名が意図を正確に表しているか
- インターフェースが呼び出し側にとって使いやすいか(引数順、戻り値、例外/エラーの扱い)
- ヘッダの依存関係が循環していないか
- TODO / 未決定箇所が明示されているか

### シニア視点でチェックすること
- スレッド境界が明確か(audio スレッド / UI スレッド / ワーカー を意識した設計か)
- DSP 数値安定性リスク(高 Q フィルタ、ゼロ除算、float 精度)
- データのオーナーシップとライフタイムが問題ないか
- パフォーマンスボトルネックになりうる設計上の問題(ヒープ割り当て、コピー、ロック)
- 拡張性:将来の変更が局所的に収まるか

### 「重大」な指摘とは
以下のいずれかに該当する場合は 2 ラウンドを超えてでも解消する:
- audio スレッドでブロッキング操作が発生する設計
- データ競合・未定義動作につながるスレッド設計
- DSP の根本的なアルゴリズム誤り
- 後続フェーズへの波及が大きいインターフェース破壊

## サイクル 2: 実装・テストレビュー

```
実装 + テスト作成
    ↓
[ラウンド 1]
  Agent起動(ジュニア視点) → 指摘リスト → 修正
  Agent起動(シニア視点)  → 指摘リスト → 修正
    ↓
[ラウンド 2 ※必要なら]
  同上(未解決 or 新たな重大指摘のみ対象)
    ↓
フェーズ完了判定 → 次フェーズへ
```

### ジュニア視点でチェックすること
- コードが設計ドキュメントと整合しているか
- 変数名・関数名が明確か
- 明らかな境界値漏れ・nullチェック漏れがないか
- テストが意図したケースを実際にカバーしているか(アサートの内容を確認)

### シニア視点でチェックすること
- audio コールバック内でのヒープ割り当て・ロック・例外がないか
- SmoothedValue / atomic / double buffer の正しい使い方になっているか
- JUCE の ValueTree / APVTS 変更通知がメッセージスレッドに閉じているか
- DSP テストが周波数応答・時定数・ナイキスト境界を定量的に検証しているか
- リソースリーク・RAII の抜けがないか

### 「重大」な指摘とは(サイクル 1 と同じ基準に加え)
- テストが通っているにもかかわらず音が明らかにおかしい(耳確認で判明)
- pluginval / auval がクラッシュする

## ラウンド打ち切りの判断基準

| 残存指摘の種類 | 対応 |
|---|---|
| 重大(上記定義に該当) | ラウンドを続ける |
| 中程度(設計意図は満たすが改善余地あり) | 次フェーズの Tech Debt として記録し、現フェーズは完了とする |
| 軽微(命名・コメント・スタイル) | その場で即修正し、追加ラウンドは不要 |

Tech Debt は `docs/TECH_DEBT.md` に記録する(フェーズ番号・内容・優先度)。

## 並列実装ルール

依存関係のないフェーズ・タスクは Agent を並列起動して実装する。詳細な依存グラフとタイムラインは [docs/PLAN.md § H](PLAN.md#h-フェーズ依存関係と並列実行戦略) を参照。

### 並列化の判断基準

| 条件 | 対応 |
|---|---|
| 入力ファイル・出力ファイルが完全に分離している | 並列起動してよい |
| 共通の基底クラスやヘッダが未完成 | 基底を先に完成させてから並列起動 |
| `PluginProcessor.cpp` など複数トラックが変更したいファイルがある | どちらかが先に終わるまでシリアライズ。またはファイル分割を検討 |
| テストが他トラックの成果物をリンクする | 依存先が `ctest` 通過してから起動 |

### レビューサイクルと並列実行の組み合わせ

- 並列トラックは**それぞれ独立してレビューサイクルを回す**(Track A のレビューを Track B が待つ必要はない)
- ただし統合フェーズ(Phase 5 統合など)に入る前に、各トラックのレビューサイクルを**完了させてから統合 Agent を起動する**

## フェーズ完了チェックリスト(共通)

PLAN.md の各フェーズ「完了の判定基準」を満たした上で:

- [ ] 設計レビューサイクル完了(最大 2 ラウンド)
- [ ] 実装・テストレビューサイクル完了(最大 2 ラウンド)
- [ ] Tech Debt があれば `docs/TECH_DEBT.md` に記録済み
- [ ] `ctest` がすべて PASS
- [ ] PLAN.md のフェーズチェックリストにチェック済み
